Section 1

Introduction

1.1 Purpose

This Software Requirements Specification defines the functional and non-functional requirements for the Cafe Vibe Finder and Recommendation Engine. The system recommends cafes based on user location, budget, cuisine preference, dietary constraints, ambience-oriented filters, and live crowd signals.

1.2 Scope

The product is a lightweight web application composed of a Java backend and a browser frontend. It ingests cafe data from two supported source formats, .csv and .xlsx, stores onboarding data in SQLite when the JDBC driver is available, and exposes recommendation, onboarding, and live-status APIs over HTTP. The frontend renders a geographic map, ranked results, and real-time interaction controls.

1.3 Intended Audience

  • Project sponsors and academic evaluators reviewing scope and quality.
  • Developers implementing and maintaining backend, data, and frontend modules.
  • Test engineers defining validation scenarios and acceptance coverage.
  • Operators or demonstrators running the system locally for review or presentation.

1.4 Definitions

Term Definition
KD-tree Spatial index used to prefilter cafes by geographic search radius.
Top-k Selection of the best k ranked results without full sorting of all candidates.
Live signal User-submitted runtime signal such as seat availability or table-sharing status.
CSV pipeline Recommendation pipeline built from the structured data/cafes.csv dataset.
XLSX pipeline Recommendation pipeline built from the Excel dataset, optionally enriched with geocoded coordinates.

1.5 References

  • Project source modules under src/ and frontend assets under web/.
  • Onboarding schema reference in docs/onboarding_schema.sql.
  • Local launcher script run.ps1 for compile-and-run execution.
Section 2

Overall Description

2.1 Product Perspective

The application is a self-contained local web system. It serves static frontend assets and REST-like HTTP APIs from a single Java process. Recommendation, onboarding, and live-status logic run in-process. Onboarding persistence is available through a local SQLite database when the SQLite JDBC dependency is present.

2.2 Product Functions

  • Load and validate cafe datasets from CSV and XLSX sources.
  • Search nearby cafes using a spatial index and Haversine refinement.
  • Collect permanent onboarding preferences and dynamic visit context.
  • Apply budget, diet, vibe, acoustic, and menu-based filters.
  • Rank candidate cafes using weighted and profile-aware scoring.
  • Display recommendations on an interactive map with result cards and map-focus actions.
  • Collect live crowd-sourced seat and table-sharing signals.

2.3 User Classes

User Class Description Primary Needs
End User User searching for cafes based on practical and vibe-oriented preferences. Fast search, understandable results, clear filters, live crowd context.
Operator Person launching the application and validating local demo behavior. Simple startup, visible health status, deterministic dataset selection.
Developer Maintainer extending ranking logic, datasets, UI, or APIs. Clear module boundaries, predictable behavior, verifiable requirements.

2.4 System Context

The platform operates as a single local application process that serves both browser assets and HTTP APIs. Users interact through a browser, the browser may supply geolocation and address resolution inputs, and the backend coordinates dataset loading, recommendation scoring, onboarding persistence, and live-status updates.

2.5 Operating Environment

  • OS: Windows with PowerShell launcher support.
  • Runtime: Java JDK for compilation and execution.
  • Browser: Modern Chromium, Edge, or Firefox-class browser.
  • Optional tooling: Python 3 with geocoding dependencies.

2.6 Design Constraints

  • No external Java Excel library is used for XLSX parsing.
  • SQLite onboarding persistence depends on the external sqlite-jdbc jar.
  • Weight inputs must sum exactly to 1.0.
  • Live seat state is stored in process memory only.
  • Frontend and backend are served from one local application.

2.7 Dependencies

  • Leaflet and marker-cluster assets loaded by the browser.
  • Browser geolocation API for live user location.
  • Nominatim lookup from browser for address-to-coordinate search.
  • Optional geocoded Excel artifact generated offline.
  • SQLite JDBC driver for onboarding persistence.
Section 3

External Interface Requirements

3.1 User Interface

The user interface shall present a map, onboarding controls, search controls, dataset-method selector, result cards, and live-signal controls. The UI shall allow the user to select either the CSV or XLSX recommendation pipeline before executing a search. The map shall display the user's resolved location using a blue marker and shall support focusing on individual recommended cafes.

3.2 Software Interfaces

Interface Type Description
GET /api/health HTTP JSON Returns service readiness and available source information.
GET /api/recommend HTTP JSON Returns ranked cafe recommendations with profile tags, explanation text, and live status.
/api/onboarding/* HTTP JSON Creates, loads, and updates onboarding profile and active visit context data.
POST /api/live/seat HTTP JSON Records seat availability crowd signals for a given cafe.
POST /api/live/table-share HTTP JSON Sets table-sharing availability signal for a given cafe.

3.3 Hardware Interfaces

No dedicated hardware interface is required beyond a general-purpose computer with network-capable browser support.

3.4 Communications Interfaces

The system communicates over HTTP on localhost. The application shall bind to the first available port in the configured startup range and display the selected port at startup.

Section 4

Functional Requirements

FR-01 Dataset Initialization

The system shall preload one recommendation pipeline for data/cafes.csv and one recommendation pipeline for the XLSX source during server startup.

FR-02 Dataset Selection

The system shall allow the user to select the recommendation source as csv or xlsx through the frontend UI and shall route the search request accordingly.

FR-03 Source Disclosure

The system shall include the selected source in recommendation responses and shall display the active source in the UI status area and result cards.

FR-04 Geographic Search

The system shall accept user latitude, longitude, and radius inputs and shall return cafes within the radius after KD-tree prefiltering and exact Haversine distance evaluation.

FR-04A Location Resolution

The system shall require a resolved user location from browser geolocation or address lookup before search execution and shall not rely on a hardcoded city-center coordinate as the effective user location.

FR-04B Onboarding Persistence

The system shall support saving and retrieving permanent onboarding preferences and the active dynamic visit context through onboarding APIs.

FR-05 Constraint Filtering

The system shall support filtering by budget, diet, menu query, vibe tags, acoustic profile, and independent-cafe preference.

FR-06 Weighted Ranking

The system shall compute a score using distance, price, rating, and cuisine-match weights and shall return top-ranked results according to ascending score.

FR-06A Profile-Aware Ranking

The system shall incorporate stored and request-time onboarding data into recommendation scoring, including user profile score, dynamic context score, category match, ambience match, budget compatibility, and distance score.

FR-07 Top-k Selection

The system shall accept a user-provided k value and return at most k ranked recommendations.

FR-08 Fallback Recommendations

If no recommendation satisfies the initial search constraints, the system shall execute a fallback search with an expanded radius and return the lowest-price subset of fallback results.

FR-09 Live Seat Voting

The system shall allow users to submit seat status as easy or standing and shall expose the aggregate counts in recommendation results and seat-status endpoints.

FR-10 Table Share Signaling

The system shall allow users to indicate whether table sharing is available for a cafe and shall expose the latest status in API responses.

FR-11 Address Lookup Trigger

The frontend shall support conversion of a free-form address to coordinates using browser-side geocoding and shall populate the search inputs with the resolved location.

FR-12 Health Monitoring

The system shall expose a health endpoint reporting service availability and supported dataset source options.

FR-12A Recommendation Explanation

The system shall return an explanation for each recommendation based on the strongest matching factors from the scoring process.

FR-13 Data Ingestion

The system shall support both CSV and XLSX ingestion and shall derive missing operational fields for XLSX rows when exact structured data is absent.

FR-14 Geocoded XLSX Preference

For the XLSX pipeline, the system shall prefer the geocoded workbook if it exists; otherwise it shall use the base workbook.

FR-15 Validation Failure Handling

The system shall reject invalid or incomplete request parameters with clear error responses and shall fail startup if no valid cafe dataset is loaded.

Section 5

Non-Functional Requirements

ID Category Requirement
NFR-01 Performance The system should avoid full dataset scans for standard searches by using spatial pruning before detailed scoring.
NFR-02 Usability The user shall be able to execute a recommendation query from the UI without interacting with backend code or configuration files.
NFR-03 Reliability The system shall return structured JSON error responses for invalid requests rather than failing silently.
NFR-04 Maintainability Core responsibilities shall remain separated across data, spatial, filter, score, rank, service, and web layers.
NFR-05 Portability The application shall run locally with Java and a browser on a standard desktop environment; onboarding persistence shall activate automatically when the SQLite JDBC dependency is available.
NFR-06 Security The system shall not execute arbitrary client input as code and shall treat query parameters as data only.
NFR-07 Observability The server shall log startup state, selected port, and selected dataset source for recommendation requests.
NFR-08 Data Integrity The system shall enforce exact weight-sum validation and reject inconsistent scoring input.
NFR-09 Explainability The recommendation response shall provide concise, human-readable reasoning tied to actual ranking signals.
Section 6

Data Requirements

6.1 Core Runtime Data

  • id, name
  • latitude, longitude
  • cuisines, avgPrice, rating
  • veg, nonVeg, vegan
  • operatingHours, address, contact

6.2 XLSX Input Mapping

  • Name, Location, Food, Ambience
  • Event, Wi-Fi, Time
  • No. of People, Speciality, MaxCapacity
  • Optional: Latitude, Longitude

6.3 Data Quality Rules

  • Datasets shall contain at least one valid cafe record after loading and validation.
  • CSV records shall provide explicit coordinates and structured attributes.
  • XLSX records may rely on derived values where source fields are incomplete.
  • Geocoded XLSX coordinates shall override approximated coordinates when present.

6.4 Retention and State

Live seat and table-share data are transient runtime state and are reset when the application restarts. When SQLite is enabled, onboarding profile data, active visit context, recommendation history, and explanation records may be persisted locally. Without the JDBC dependency, onboarding falls back to browser-local storage only.

Section 7

Acceptance Criteria

ID Scenario Acceptance Condition
AC-01 Startup Running .\run.ps1 starts the application on a free local port and prints health and dataset information.
AC-02 Dataset selection Changing the UI source selector from CSV to XLSX changes the source value returned by /api/recommend.
AC-03 Recommendation output Each successful recommendation response includes result count, selected source, ranked items, profile tag, explanation text, insights, and live status fields.
AC-04 Seat voting Submitting a seat vote updates the corresponding cafe live-status values on the next query.
AC-05 Input validation Invalid source names or malformed numeric parameters return a client error response with a readable message.
AC-06 Fallback behavior When no cafes satisfy the initial criteria, the system returns a non-crashing fallback set when qualifying cafes exist in the expanded radius.
AC-07 Location handling The system does not execute a search until location is captured from browser geolocation or resolved from a user-provided address.
AC-08 Map interaction Recommended cafes appear on the map, the map fits the visible recommendation set, and each result card can focus its cafe marker on the map.
Section 8

Risks and Assumptions

8.1 Assumptions

  • The user operates the system on a machine capable of running Java and a modern browser.
  • The browser permits geolocation when the live-location feature is used.
  • The datasets remain small enough for local-memory operation without external storage.
  • The project is currently executed in a local or demo environment rather than a hardened production deployment.

8.2 Key Risks

  • Approximate XLSX coordinates may reduce recommendation accuracy if geocoding has not been completed.
  • In-memory live state may be lost on restart and is unsuitable for production persistence requirements.
  • SQLite dependency gap leaves onboarding persistence disabled until the JDBC jar is added to the local runtime.
  • Strict weight validation may reject user input if UI-supplied values do not sum exactly to 1.0.
  • External geocoding dependency may be limited by network or upstream service constraints during data enrichment.
